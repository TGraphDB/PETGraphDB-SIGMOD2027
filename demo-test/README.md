# PETGraphDB VLDB 2026
> PETGraphDB is a database system designed for managing temporal graph data. 

## Purpose of `demo-test`

`demo-test` is the **testing and benchmarking module** for PETGraphDB.
Its primary role is to run controlled experiments and compare PETGraphDB with other database systems under the same workload, rather than to provide PETGraphDB core database engine code.

Specifically, this module is used to:

- Prepare and load temporal graph datasets for repeatable experiments.
- Execute PETGraphDB application workloads (read, write, and analytical transactions).
- Collect execution metrics such as latency, throughput/concurrency, and storage-related indicators.
- Run side-by-side evaluations against comparison systems (Neo4j, PostgreSQL, MariaDB) under aligned settings.
- Support automated experiment orchestration through Docker/Jenkins-related scripts.

In short, if `temporal-storage` and `temporal-neo4j-4.4` implement PETGraphDB itself, `demo-test` is the project that **tests and validates PETGraphDB performance and behavior**.

We develop a set of experiments to evaluate the performance of PETGraphDB in typical scenarios of temporal property graph data, mainly from three aspects: 
disk space utilization, latency of read and write operations, and transaction throughput and concurrency.

# Datasets
Three datasets are used in the experiments: two real-world temporal graph datasets and one synthetic dataset.

- **Energy**: European renewable energy power system dataset.
- **Traffic**: Beijing road traffic temporal graph dataset.
- **SYN**: synthetic temporal graph dataset generated by the benchmark data generator.

The temporal property **non-redundancy** metric is the average probability that a temporal-property update event changes the state value (`V` for node temporal properties and `E` for edge temporal properties).

# Systems
We compare PETGraphDB (implemented on Neo4j v4.4.37, denoted as **PETG**) with four baseline systems that can support PETG application-level ACID workloads.

- **Nepal**: temporal relational-database style solution (transaction-time native). Because source code is unavailable, we implement a valid-time version using MariaDB (11.4.7). Temporal properties are modeled as `<vertex|edge, τs, τe, tp1, ..., tpn>`, and indexes on `<vertex|edge, τs>` are used to accelerate *Entity-History*.
- **TGQL**: graph-based temporal modeling approach that attaches valid time intervals to graph entities. Since source code is unavailable, we re-implement TGQL on Neo4j 4.4.37.
- **AeonG**: temporal graph database built on Memgraph + RocksDB, natively transaction-time only. We map dataset valid-time into system transaction-time and preprocess requests accordingly for fair comparison.
- **Aion**: temporal graph database built on Neo4j, natively transaction-time only. The same valid-time to transaction-time conversion protocol is used as in AeonG.

Temporal graph analytic systems that do not support full user-level ACID transactions are not included in this benchmark.

# Environment
All experiments are conducted on a Dell T630 workstation with dual Intel Xeon E5-2640 v3 @ 2.60GHz (16/32 physical/logical cores), 192GB RAM, and Dell PERC H730 RAID1 (SAS 7200 rpm, 8TB) storage.

- OS: Ubuntu 24.04.2 LTS
- JDK: OpenJDK 11.0.27+6
- Default memory allocation per evaluated system: 48GB

For SQL-based baselines in this benchmark project, default transaction isolation is `read-committed`.

For the storage-space import stress setting reported in the paper, 185GB memory is allocated per system during import; in that setting, Aion fails on Traffic and SYN and AeonG fails on SYN due to OOM.

# Data loading and applications
Data loading is implemented by system-specific bulk loaders in `src/main/java/edu/buaa/batch`.

- **PETG**: static topology + temporal properties stored with PETGraphDB temporal storage engine.
- **Neo4j/TGQL-style loaders**: use valid-time interval modeling on graph entities.
- **Nepal(MariaDB) implementation**: static tables for nodes/edges, and valid-time temporal tables for temporal properties.
- **Aion/AeonG adapters**: workloads are preprocessed by valid-time to transaction-time conversion for compatibility.

Applications (queries/transactions) used in experiments:

- **Read operations**: `Entity-History`, `Snapshot`, `GATP(max)`, `ETPC`
- **Write operations**: `Append`, `Update`
- **Temporal graph analytic query**: `Reachable Area` (Traffic dataset)

Evaluation protocols aligned with the paper:

- **Latency evaluation**: for each dataset and operation, random 10k requests, single-thread execution, fully loaded database; metric is 90th-percentile latency. Each `Append` and `Update` request contains 100 operations.
- **HTAP throughput evaluation**: 100k mixed requests with ratio `Append 40%`, `Update 5%`, `Entity-History 35%`, `Snapshot 5%`, `GATP(max) 5%`, `ETPC 5%`, `Reachable 5%`; maximum concurrent connections = 16.
- **Space evaluation**: full dataset loaded and measured by disk footprint excluding transaction logs.

# Code Architecture
`src/main/java/` contains most of the code that implemented the applications and performed the test. some of the important folders are:
- `batch`: import datasets into applications.
- `client` & `server`: application database adapters.
- `dataset`: dataset adapters.
- `common/transaction`: application query objects.
- `common/benchmark`: generate & run benchmark queries on applications.

Each query is encapsulated into a transaction. Each transaction has some parameters, the result set of the query, and the metrics generated during execution, such as the execution time, and whether it failed.

Some other codes not very related:
- `docker`: build the PETGraphDB docker image file.
- `jenkins`: jenkins procedurces that deploy and run benchmark on client & server machines.
- `server`: operation dashboard of all test cases, results, machines, application status.
