# PETGraphDB VLDB 2026
> PETGraphDB is a database system designed for managing temporal graph data. 

## Purpose of `demo-test`

`demo-test` is the **testing and benchmarking module** for PETGraphDB.
Its primary role is to run controlled experiments and compare PETGraphDB with other database systems under the same workload, rather than to provide PETGraphDB core database engine code.

Specifically, this module is used to:

- Prepare and load temporal graph datasets for repeatable experiments.
- Execute PETGraphDB application workloads (read, write, and analytical transactions).
- Collect execution metrics such as latency, throughput/concurrency, and storage-related indicators.
- Run side-by-side evaluations against comparison systems (Neo4j, PostgreSQL, MariaDB) under aligned settings.
- Support automated experiment orchestration through Docker/Jenkins-related scripts.

In short, if `temporal-storage` and `temporal-neo4j-4.4` implement PETGraphDB itself, `demo-test` is the project that **tests and validates PETGraphDB performance and behavior**.

We develop a set of experiments to evaluate the performance of PETGraphDB in typical scenarios of temporal property graph data, mainly from three aspects: 
disk space utilization, latency of read and write operations, and transaction throughput and concurrency.

# Datasets
Three datasets are used for evaluation. Two real-world temporal graph datasets (Beijing road traffic data, referred to as Traffic, and European renewable energy power system data, referred to as Energy) 
and a synthetic temporal graph data (SYN) generated by a temporal graph data generator. Some statistical indicators of the datasets are listed in Table.
The `temporal property non-redundancy rate` is the average probability that update events of temporal properties in a dataset lead to state changes (`V` for temporal properties on nodes, `E` for that on edges). 
This metric characterizes how much data can be considered semantically non-redundant.


# Systems
We compared PETGraphDB with relational databases (PostgreSQL 17.5, referred to as PG), temporal relational databases (MariaDB 11.4.7, referred to as MA), and graph databases (Neo4j 4.4.37). 
These databases were chosen because they support transactions, application-time data model, and are currently the common solutions for temporal graph data management. 
Other temporal graph systems are not considered because they either only native support transaction-time temporal graph data (Aion from Neo4j Research, AeonG from Renmin University China) or 
do not support full user-level transactions.

# Environment
The experiment was conducted on a Dell T630 workstation with Intel(R) Xeon(R) CPU E5-2640v3@2.60GHz (x2) and 192GB memory. 
The storage is a Dell PERC H730 Adp RAID1 disk array (SAS 7200 rpm, 8TB).
The operating system is Ubuntu 24.04.2 LTS, with OpenJDK (version 11.0.27+6) installed.
For each system, we allocate 48GB memory buffer/cache (`Xmx=48g` for systems based on JVM) and 
use the `read-committed` isolation level for transactions by default.

# Data loading and applications
When loading data on Neo4j, we create nodes that represent a time interval and link these nodes with entities in the temporal graph. 
For PG, we create a table to store the nodes with their static properties and a table to store the relationships with their static properties. 
The temporal properties of nodes/rels are stored in corresponding tables in the form of `nodeId/relId, timeStart, tp_1, ..., tp_n`. 
For MariaDB, application-time temporal tables are used to store the temporal property data in the form of `nodeId/relId, timeStart, timeEnd, tp_1, ..., tp_n`. 

For each dataset, we constructed a set of applications and implemented these applications on both PETGraphDB and the comparison systems. 
These applications are four read operations Entity-History, Snapshot, ATP(max), and ETPC, two write operations Append and Update, and a typical temporal graph analytical query named Reachable Area.

Note that for PG and MariaDB, indexes on `nodeId/relId, timeStart` are created to accelerate the Entity-History query. 
The typical queries are selected to test the performance of tgraph Temporal Graph Database System in specific application scenarios (and compared with neo4j, MariaDB and PostgreSQL).

# Code Architecture
`src/main/java/edu/buaa` contains most of the code that implemented the applications and performed the test. some of the important folders are:
- `batch`: import datasets into applications.
- `client` & `server`: application database adapters.
- `dataset`: dataset adapters.
- `common/transaction`: application query objects.
- `common/benchmark`: generate & run benchmark queries on applications.

Each query is encapsulated into a transaction. Each transaction has some parameters, the result set of the query, and the metrics generated during execution, such as the execution time, and whether it failed.

Some other codes not very related:
- `docker`: build the PETGraphDB docker image file.
- `jenkins`: jenkins procedurces that deploy and run benchmark on client & server machines.
- `server`: operation dashboard of all test cases, results, machines, application status.
